# Technical Concerns

*Last updated: 2026-06-10 — reflects WhatsApp-only cleanup state, validated against codebase*

## Critical Issues (Security, Data Loss, Correctness)

### C1. CORS Wildcard in Production (RESOLVED)
Restricted exclusively to `process.env.FRONTEND_URL` during the Zero-Trust refactor.

### C2. CSP Allows `unsafe-inline` and `unsafe-eval` (RESOLVED)
CSP headers strictly hardened. `unsafe-inline` completely purged from Helmet config.

### C3. Credential Encryption Fallback Key (RESOLVED)
AES-256-GCM enforced. System fatally crashes on boot if `CREDENTIAL_ENCRYPTION_KEY` is not precisely 32 bytes.

### C4. DEV_AUTH_BYPASS Leakage Risk (RESOLVED)
Legacy bypass code deleted entirely. Clerk JWT authentication is strictly enforced system-wide.

### C5. API Key Stored as SHA-256 Hash
`src/middleware/auth.ts:45` — API keys are hashed with plain SHA-256 (not HMAC, no pepper). Since API keys are generated by the system, the keys themselves are the secret. SHA-256 is fast to brute-force. Consider HMAC-SHA256 with a server-side pepper.

### C6. Stale Error Code Prefixes for Deleted Platforms
`src/errors/codes.ts:56-70` — `TG_001..TG_003` (Telegram) and `TW_001..TW_10` (Twitter) error codes remain in the registry despite those platforms being deleted. Dead code that adds confusion and inflates the error surface.

### C7. Stale Test Files for Deleted Platforms
`src/__tests__/` contains 9 test files referencing deleted platform code (discord-*.test.ts, telegram-webhook.test.ts, integration/*.test.ts). These will fail at import time and pollute test results.

## Technical Debt

### TD1. Single-File AI Orchestrator (RESOLVED)
Refactored to Unified OpenRouter Factory. The 13 legacy providers have been purged, simplifying the orchestration bottleneck.

### TD2. Static Adapter Methods
`src/adapters/whatsapp.adapter.ts` — `WhatsAppAdapter` uses all-static methods, making it impossible to mock the underlying `sendText`/`getConnectionState` calls in tests without patching the entire evolutionApi module.

### TD3. Inconsistent Error Response Format
Routes return errors in at least 3 patterns:
- `createAppError(ErrorCode.SYS_003, 'Internal Server Error')` (credentials)
- `{ error: 'msg', details: ... }` (whatsapp-chat)
- Plain string errors in some middleware

No API-wide error response envelope.

### TD4. AI Client Caching Without Invalidation (RESOLVED)
Transitioned to stateless OpenRouter fetch mechanics. Legacy singletons purged.

### TD5. Empty Catch Blocks Across Source
128 catch blocks found with `grep`. While many are intentional (fallback to defaults), at least 14 are completely empty `catch {}` blocks that silently swallow errors without logging.

### TD6. `as any` Type Assertions
Heavy use of `as any` throughout:
- Gateway routes: `(req as any).params?.tenantId`
- WhatsApp chat: `(req as any).user`
- Quota middleware: `(req as any).params?.tenantId`

This bypasses Express type safety and hides bugs when route signatures change.

### TD7. Frontend API Service Monolith
`frontend/src/services/api.ts` is the single gateway for all backend communication. As conversation/message features grow, this file becomes a bottleneck for maintenance and tree-shaking.

## Missing Features / Gaps

### G1. No WebSocket/SSE for Real-time Messages (Partial)
WhatsApp message delivery uses Evolution API webhooks → queue → worker → Socket.IO emit. This works but has no guaranteed delivery — if the Socket.IO client is disconnected when the worker emits, the message is lost until manual refresh. No message replay or catch-up mechanism.

### G2. No Rate Limiting on Socket.IO Connections
Webhooks are rate-limited (100/sec per tenant), but Socket.IO connections have no rate limiting or connection cap per tenant.

### G3. No Audit Trail for Sensitive Operations (RESOLVED)
The `AuditLog` PostgreSQL table has been established. Destructive actions (like bot deletion) now permanently record the user ID and timestamp to the un-editable audit registry.

### G4. No Input Sanitization on AI Prompts
User messages flow directly into the AI orchestrator and are sent to third-party LLM APIs. No sanitization or prompt injection filtering.

### G5. Missing Health Check Dashboard
No admin page or endpoint to view system health: queue depth, Redis connectivity, adapter circuit breaker states, API quota usage per tenant. The `/health` and `/ready` endpoints are minimal.

### G6. Settings Page Tab State Persistence
`frontend/src/pages/SettingsPage.tsx` — All 6 tabs are mounted but only the active one renders (conditional rendering, not unmounting). This means form state persists across tab switches, which may cause stale state bugs.

## Dependency Risks

### D1. express-rate-limit v7 (ESM/CJS)
Phase 18 feedback documented that v8+ is ESM-only. Current usage works with v7 but upgrading Express or related middleware may break.

### D2. BullMQ Redis Dependency
The entire message pipeline depends on Redis. No fallback or graceful degradation if Redis is down — WhatsApp messages will silently fail.

### D3. Evolution API External Dependency
All WhatsApp functionality depends on a third-party Evolution API service. No SLA, no local message store for offline delivery, no queue retry dead-letter handling.

### D4. @google/generative-ai SDK
Direct SDK import alongside the OpenAI-compatible pattern used for other providers. Creates inconsistency in error handling and streaming formats.

## Performance Concerns

### P1. AI Orchestrator: N+1 Bot Lookup
`src/ai/orchestrator.ts:33` — `prisma.bot.findFirst` is called on every AI request. No caching. Under load this will create unnecessary DB reads.

### P2. Quota Check: Extra DB Read Per Request
`src/middleware/quota.ts:28` — `prisma.tenant.findUnique` runs on every authenticated request to get the plan. Should be cached in Redis alongside the quota counter.

### P3. No Database Connection Pool Tuning
No explicit `connection_limit` in the Prisma connection URL. Default pool size may be insufficient under load.

## Reliability Risks

### R1. No Dead-Letter Queue for Failed WhatsApp Messages
Messages that fail after retries are silently dropped. The DLQ monitor (`src/workers/dlq.ts`) exists but there's no reprocessing capability or alerting.

### R2. Circuit Breaker State In-Memory Only
`src/utils/httpProxy.ts` — Circuit breaker state is stored in a `Map`. On process restart, all circuit breakers reset to `closed` regardless of upstream health.

### R3. Email Service Plain-Text Fallback
`src/utils/email.ts` — Falls back to `"Email sent (development mode — no actual email sent)"` in development. If `NODE_ENV` is misconfigured, production emails silently fail.

## Maintainability Concerns

### M1. Stale Platform Code in Error Registry
`src/errors/codes.ts` still contains `TG_*` (Telegram) and `TW_*` (Twitter) error codes. These should be removed to avoid confusion.

### M2. Stale Test Files
9 test files in `src/__tests__/` reference deleted platform code. These should be removed or they'll cause test suite failures.

### M3. Settings Page Tab Architecture
`frontend/src/pages/SettingsPage.tsx` — All 6 tabs are mounted but only the active one renders. Form state persists across tab switches, which may cause stale state bugs.

### M4. Prisma Schema Has Unmigrated Changes
Missing `User.name` column caused backend crashes in dev bypass mode. Migrations must be deployed and synced with Prisma schema to avoid further `Invalid prisma.user.findFirst()` crashes. Currently mitigated via mocked JSON user.

## Scalability Bottlenecks

### S1. Single-Process Architecture
No clustering, no worker separation. Both HTTP serving and BullMQ workers run in the same process. Under WhatsApp message spikes, API response times will degrade.

### S2. No Message Persistence for WhatsApp (Partial)
Messages are persisted to PostgreSQL, but the outbound send is fire-and-forget. If the process crashes between DB write and Evolution API send, the message is marked as sent but never delivered.

### S3. In-Memory Rate Limiter State
Webhook rate limiting (100/sec per tenant) uses Redis, but the per-IP `express-rate-limit` uses `RedisStore` which has known race conditions under high concurrency.

## Test Coverage Gaps

### T1. WhatsApp Chat Route Untested
`src/routes/whatsapp-chat.ts` — no test file exists. Core endpoints (`/chats`, `/contacts`, `/messages/:jid`, `/send`) have zero coverage.

### T2. AI Orchestrator Untested
`src/ai/orchestrator.ts` — the core business logic for message processing has no test file. Provider resolution, quota enforcement, and error handling are all untested.

### T3. Frontend Component Tests
~90 TSX/TS files in frontend, zero component test files (`*.test.tsx`). Settings tabs, bot management, conversation views — all untested.

### T4. Stale Test Files Mask Real Coverage
9 test files reference deleted platform code. These will fail at import time, making the test suite appear worse than it is. Should be removed to get accurate coverage metrics.
