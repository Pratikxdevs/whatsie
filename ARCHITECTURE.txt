1. SYSTEM OVERVIEW
================================================================================

PURPOSE
-------
Multi-tenant SaaS platform enabling businesses to deploy AI-powered bots across
8 messaging platforms (WhatsApp, Telegram, Discord, FB Messenger, Instagram, 
MS Teams, Twitch, Twitter/X).

CORE CAPABILITIES
-----------------
- Unified message handling across all platforms
- AI-powered conversation with cost controls
- CRM integration for lead management
- Workflow automation engine
- Real-time dashboard and analytics
- Per-tenant customization and billing

KEY DESIGN PRINCIPLES
---------------------
1. Platform-agnostic core (never couple to specific APIs)
2. Multi-tenancy from Layer 0 (tenant isolation is foundational)
3. Queue-driven architecture (survive spikes and API failures)
4. AI as a tool, not the brain (rules/workflows before LLM calls)
5. Bidirectional normalization (parse inbound, render outbound)

3. LAYER-BY-LAYER ARCHITECTURE
================================================================================

LAYER 0: MULTI-TENANCY & ISOLATION
-----------------------------------
PURPOSE: Ensure every request is scoped to a tenant. Prevents data leaks.

Components:
  - Tenant Router
    * Extracts tenantId from API key or JWT
    * Injects into request context
    * Validates tenant exists and is active

  - Tenant Context Middleware
    * Sets AsyncLocalStorage context
    * All downstream code reads tenantId from context
    * Database queries auto-filter by tenantId

  - Tenant Config Store (Redis)
    * Per-tenant settings: AI model, rate limits, workflows
    * Cache with 5min TTL, fallback to PostgreSQL

  - Row-Level Security (PostgreSQL)
    * Every table has tenantId column
    * RLS policies enforce filtering
    * Example: CREATE POLICY tenant_isolation ON messages 
      USING (tenantId = current_setting('app.tenantId')::uuid)

Critical Rules:
  - NEVER allow tenantId to be set by user input
  - ALWAYS derive tenantId from authenticated credentials
  - Database queries MUST include WHERE tenantId = $1
  - Test with 2+ tenants to verify isolation

LAYER 1: PLATFORM API ADAPTERS
-------------------------------
PURPOSE: Abstract platform-specific APIs behind unified interface.

Components per Platform:

  WhatsApp Adapter:
    - Evolution API Docker container on localhost:8080
    - Send: POST /message/sendText/{instanceId}
    - Receive: Webhook at /gateway/whatsapp
    - Health check: GET /instance/connectionState/{instanceId}

  Telegram Adapter:
    - tdlib/telegram-bot-api Docker on localhost:8081
    - Send: POST /bot{token}/sendMessage
    - Receive: Webhook at /gateway/telegram
    - Long polling fallback if webhook fails

  Discord Adapter:
    - discord.js Client with WebSocket gateway
    - Send: channel.send() or user.send()
    - Receive: Events via 'messageCreate' listener
    - Reconnection logic with exponential backoff

  Meta Adapter (FB + Instagram):
    - Direct HTTPS to graph.facebook.com/v25.0
    - Send: POST /{pageId}/messages
    - Receive: Webhook at /gateway/meta (handles both)
    - Token: Page Access Token (never-expiring)
    - Constraint: 24-hour messaging window

  Teams Adapter:
    - BotFrameworkAdapter from botbuilder
    - Send: context.sendActivity()
    - Receive: Bot handles POST /api/messages
    - Direct Line for WebSocket fallback

  Twitch Adapter:
    - @twurple/api, @twurple/eventsub-ws, @twurple/chat
    - Send: chatClient.say()
    - Receive: EventSub WebSocket + Chat IRC
    - Separate connections for events vs chat

  Twitter Adapter:
    - twitter-api-v2 with OAuth 2.0
    - Send: client.v2.tweet() or client.v2.sendDM()
    - Receive: Webhook (paid tier) or polling
    - Rate limit: Strict, see Layer 8

Interface Contract:
  All adapters implement:
    - sendMessage(recipient, content, options)
    - parseIncomingWebhook(req.body) → NormalizedMessage
    - verifySignature(req) → boolean
    - healthCheck() → { status, latency }

Critical Rules:
  - Adapters are STATELESS
  - No business logic inside adapters
  - Errors bubble up to gateway layer
  - Retry logic handled by queue, not adapter

LAYER 2: WEBHOOK GATEWAY
-------------------------
PURPOSE: Single secure entry point for all platform webhooks.

Components:

  Signature Validator:
    - WhatsApp: HMAC-SHA256 with Evolution API secret
    - Telegram: SHA256 hash of bot token
    - Discord: Ed25519 signature verification
    - Meta: SHA256 HMAC with app secret
    - Teams: JWT validation with bot credentials
    - Twitch: HMAC-SHA256 with secret
    - Twitter: HMAC-SHA256 with consumer secret

  Tenant Authentication:
    - Extract API key from X-API-Key header or webhook path
    - Lookup tenantId from api_keys table
    - Reject if tenant is suspended or over quota
    - Rate limit per tenant: 100 webhooks/second

  Idempotency Layer (Redis):
    - Key pattern: idempotency:{platform}:{messageId}
    - Value: "processed"
    - TTL: 24 hours
    - Check: IF EXISTS, return 200 immediately
    - Set: After normalization, before queueing

  Request Logger:
    - Log every webhook with:
      * Timestamp, tenantId, platform, messageId
      * Processing time, status code
      * Error details if failed
    - Forward to observability stack

Endpoints:
  POST /gateway/whatsapp     → WhatsApp Evolution API webhook
  POST /gateway/telegram     → Telegram Bot API webhook
  POST /gateway/discord      → Discord interactions endpoint
  POST /gateway/meta         → Meta (FB + Instagram) webhook
  POST /api/messages         → MS Teams Bot Framework webhook
  POST /gateway/twitch       → Twitch EventSub webhook
  POST /gateway/twitter      → Twitter Account Activity webhook

  GET /gateway/:platform     → Verification endpoint (Meta, Telegram)

Critical Rules:
  - ALWAYS verify signature before processing
  - Return 200 OK within 3 seconds (queue async processing)
  - Never trust message content for tenant identification
  - Log failed signature validations (possible attack)

LAYER 3: MESSAGE NORMALIZER (BIDIRECTIONAL)
--------------------------------------------
PURPOSE: Convert platform payloads to/from unified internal format.

Unified Message Schema:
  {
    tenantId: string (UUID)
    platform: 'whatsapp' | 'telegram' | 'discord' | 'messenger' | 'instagram' | 'teams' | 'twitch' | 'twitter'
    userId: string (platform-specific ID, e.g. PSID, Discord snowflake)
    conversationId: string (thread/channel ID)
    message: {
      text: string | null
      attachments: Array<{
        type: 'image' | 'video' | 'audio' | 'file'
        url: string
        mimeType: string
        size: number
      }>
      quickReplies: Array<{ label: string, payload: string }> | null
    }
    type: 'text' | 'image' | 'video' | 'audio' | 'file' | 'location' | 'contact' | 'sticker' | 'button_click'
    timestamp: ISO8601 string
    metadata: {
      raw: object (original platform payload)
      replyTo: string | null (message ID being replied to)
      isForwarded: boolean
      mentions: string[] (user IDs mentioned)
    }
  }

Inbound Parser:
  - Accepts raw webhook payload per platform
  - Extracts core fields into unified schema
  - Downloads attachments to temp storage (S3/local)
  - Stores raw payload in metadata.raw for debugging
  - Returns NormalizedMessage

Outbound Renderer:
  - Accepts unified response format
  - Converts to platform-specific payload
  - Maps rich content:
    * Buttons → WhatsApp list messages
    * Buttons → Telegram inline keyboards
    * Buttons → Discord components
    * Buttons → Meta quick replies
  - Handles fallbacks when feature unsupported
  - Returns platform-ready payload

Platform-Specific Mappings:

  WhatsApp (Evolution API):
    Inbound: webhook.data.message.conversation → message.text
             webhook.data.message.imageMessage → attachment
    Outbound: text → {number, text}
              buttons → {number, buttonMessage: {text, buttons}}

  Telegram:
    Inbound: update.message.text → message.text
             update.message.photo → attachment
    Outbound: text → {chat_id, text}
              buttons → {chat_id, text, reply_markup: {inline_keyboard}}

  Discord:
    Inbound: message.content → message.text
             message.attachments → attachments array
    Outbound: text → {content}
              embeds → {embeds: [{title, description, fields}]}

  Meta (FB + Instagram):
    Inbound: messaging[0].message.text → message.text
             messaging[0].message.attachments → attachments
    Outbound: text → {recipient: {id}, message: {text}}
              buttons → {message: {attachment: {type: template, buttons}}}

  Teams:
    Inbound: activity.text → message.text
             activity.attachments → attachments
    Outbound: text → {type: message, text}
              cards → {type: message, attachments: [AdaptiveCard]}

  Twitch:
    Inbound: event.message.text → message.text
    Outbound: text → chatClient.say(channel, text)

  Twitter:
    Inbound: direct_message_events[0].message_create.message_data.text → text
    Outbound: text → {event: {type: message_create, message_data: {text}}}

Critical Rules:
  - NEVER lose information during normalization
  - metadata.raw preserves original payload for debugging
  - Outbound rendering must handle missing features gracefully
  - Test with real webhook payloads from each platform

LAYER 4: QUEUE LAYER (BullMQ + Redis)
--------------------------------------
PURPOSE: Buffer message processing, survive spikes, enable retries.

Queue Topology:

  Per-Platform Queues:
    - whatsapp-messages
    - telegram-messages
    - discord-messages
    - meta-messages (FB + Instagram combined)
    - teams-messages
    - twitch-messages
    - twitter-messages

  Priority Lanes (within each queue):
    - VIP users: priority 1
    - Standard: priority 5
    - Bulk/automated: priority 10

  Special Queues:
    - scheduled-messages (delayed jobs for follow-ups)
    - webhook-retries (failed webhook deliveries)
    - ai-requests (long-running AI calls, separate pool)

Job Schema:
  {
    id: string (auto-generated)
    tenantId: string
    platform: string
    normalizedMessage: NormalizedMessage
    priority: number
    attempts: number
    timestamp: number
    metadata: {
      queuedAt: timestamp
      retryCount: number
      originalJobId: string (if retry)
    }
  }

Queue Configuration:

  Concurrency:
    - whatsapp-messages: 50 workers
    - telegram-messages: 100 workers
    - discord-messages: 200 workers (highest throughput)
    - meta-messages: 30 workers (rate limited)
    - others: 50 workers

  Retry Strategy:
    - Max attempts: 5
    - Backoff: exponential (1min, 2min, 4min, 8min, 16min)
    - Conditions:
      * Retry on network errors
      * Retry on 429 rate limit (after backoff)
      * Retry on 5xx server errors
      * NO retry on 4xx client errors (except 429)

  Dead Letter Queue (DLQ):
    - After 5 failed attempts → move to dlq-{platform}
    - DLQ never auto-retries
    - Manual replay via admin dashboard
    - Alert on DLQ depth > 100

  Job Cleanup:
    - Completed jobs: retain 24 hours then delete
    - Failed jobs: retain 7 days then delete
    - DLQ: retain 30 days

Redis Configuration:
  - Persistence: RDB snapshot every 5 minutes + AOF
  - Memory policy: allkeys-lru
  - Dedicated Redis instance for queues (not shared with cache)
  - Cluster mode if handling >100K messages/day

Critical Rules:
  - NEVER process messages synchronously in webhook handler
  - Queue immediately, respond 200 OK, process async
  - Monitor queue depth per platform
  - Alert if queue depth > 10K or processing time > 30s

LAYER 5: WORKER POOL
---------------------
PURPOSE: Process queued messages through business logic pipeline.

Worker Process:
  - Stateless Node.js processes
  - Pull jobs from BullMQ
  - Can scale horizontally (10, 50, 100+ instances)
  - Each worker handles one message at a time
  - Kubernetes HPA scales based on queue depth

Processing Pipeline (per message):

  1. Session Manager
     - Load from Redis: session:{tenantId}:{userId}
     - Schema: {
         conversationState: 'idle' | 'in_flow' | 'waiting_input'
         currentWorkflowId: string | null
         currentStep: number
         variables: Record<string, any>
         lastMessageAt: timestamp
       }
     - TTL: 24 hours of inactivity
     - Context Window: Load last 10 messages from PostgreSQL

  2. Intent Classifier
     - Fast, cheap layer before AI
     - Methods:
       a) Regex patterns: "price", "cost" → intent:pricing
       b) Keyword matching: "stop", "unsubscribe" → intent:optout
       c) Button payloads: payload includes intent
       d) Small ML model (optional): DistilBERT fine-tuned
     - Returns: { intent: string, confidence: number }
     - Threshold: confidence > 0.8 to skip AI

  3. Rule Engine
     - Hard-coded business rules
     - Examples:
       * If intent:pricing → send pricing message
       * If intent:optout → mark user opted_out, stop processing
       * If message contains "human" → route to agent
     - Rules stored in PostgreSQL, cached in Redis
     - Returns: { handled: boolean, response?: string }

  4. Workflow Engine
     - Check if user is mid-workflow
     - Load workflow definition from DB
     - Workflow schema:
       {
         id: string
         name: string
         trigger: { type: 'keyword' | 'intent' | 'event', value: string }
         steps: Array<{
           id: string
           type: 'message' | 'wait' | 'condition' | 'api_call' | 'update_lead'
           config: object
           nextStep: string | null
         }>
       }
     - Execute current step
     - Update session with next step
     - Returns: { handled: boolean, response?: string }

  5. AI Orchestrator (if rules + workflows didn't handle)
     - See Layer 6 for details

  6. CRM Writer
     - Update lead status
     - Log event in events table
     - Update conversation last_message_at
     - See Layer 7 for details

Output:
  - Response object: { text, attachments?, buttons?, quickReplies? }
  - Updated session state
  - CRM updates (lead stage, events)

Worker Health:
  - Expose /health endpoint
  - Prometheus metrics: jobs_processed, processing_time, errors
  - Graceful shutdown on SIGTERM (finish current job)

Critical Rules:
  - Workers are STATELESS (session in Redis, not memory)
  - No worker-to-worker communication
  - Failed jobs go back to queue for retry
  - Log every step with tenantId + userId for tracing

LAYER 6: AI ORCHESTRATION LAYER
--------------------------------
PURPOSE: Control AI usage, costs, and quality.

Components:

  Cost Controller:
    - Per-tenant token usage tracking (PostgreSQL)
    - Redis counter: tokens_used:{tenantId}:{month}
    - Hard limits per plan:
      * Free tier: 10K tokens/month
      * Pro tier: 100K tokens/month
      * Enterprise: unlimited
    - If limit exceeded: fallback to canned responses
    - Alert tenant at 80% usage

  Model Router:
    - Strategy: cheap → expensive
    - Tier 1: GPT-3.5-turbo (cheap, fast)
      * Use for: simple questions, FAQs
      * Token limit: 500 input + 150 output
    - Tier 2: GPT-4-turbo (expensive, smart)
      * Use for: complex requests, escalations
      * Token limit: 2000 input + 500 output
    - Tier 3: Claude Opus (most expensive, best reasoning)
      * Use for: critical decisions, sentiment analysis
    - Decision logic:
      if (message.length < 100 && !requiresContext) → tier1
      else if (conversationHistory.length < 5) → tier1
      else → tier2

  Prompt Builder:
    - System prompt template:
      ```
      You are a helpful assistant for {businessName}.
      Current lead: {leadName}, status: {leadStage}
      Context: {lastNMessages}
      Goal: {currentWorkflowGoal}
      Constraints: {responseConstraints}
      ```
    - Inject:
      * Tenant business info
      * Lead data from CRM
      * Conversation history (last 5 turns)
      * Current workflow goal
      * Response constraints (length, tone, etc.)

  RAG Engine (optional):
    - Vector DB: Pinecone or Qdrant
    - Embed tenant knowledge base:
      * FAQs
      * Product docs
      * Past successful responses
    - On query:
      1. Embed user message
      2. Similarity search (top 3 docs)
      3. Inject into prompt context
    - Fallback: If no relevant docs, use LLM only

  Guardrails:
    - Content filter: OpenAI moderation API
    - PII scrubber: Regex + NER model
      * Remove: credit cards, SSN, API keys
      * Mask: phone numbers (show last 4)
    - Compliance check:
      * No medical advice
      * No financial advice
      * No legal advice
    - Fallback: If violation detected, send canned response

  Response Post-Processor:
    - Length check: Max 2000 chars (WhatsApp limit)
    - Format: Markdown → platform-specific
    - Link extraction: Shorten URLs if needed
    - Emoji normalization

AI Call Flow:
  1. Check if user message passes guardrails
  2. Check tenant token quota
  3. Select model tier
  4. Build prompt with context + RAG
  5. Call LLM API
  6. Log usage: tokens, cost, latency
  7. Post-process response
  8. Return to worker

Error Handling:
  - Rate limit (429): Exponential backoff, max 3 retries
  - Timeout: 30s max, fallback to canned response
  - Content violation: Log + fallback
  - API error: Retry once, then fallback

Monitoring:
  - Metrics: tokens_per_request, cost_per_tenant, latency
  - Alerts: cost spike, error rate >5%, latency >10s

Critical Rules:
  - NEVER call AI without checking quota
  - ALWAYS log token usage for billing
  - Fallback to canned responses on failure
  - Never expose raw API errors to user

LAYER 7: CRM INTEGRATION LAYER
-------------------------------
PURPOSE: Persist lead data, conversation history, analytics.

Data Models (PostgreSQL):

  tenants:
    - id (UUID, PK)
    - name (string)
    - plan ('free' | 'pro' | 'enterprise')
    - status ('active' | 'suspended' | 'cancelled')
    - createdAt, updatedAt

  users:
    - id (UUID, PK)
    - tenantId (UUID, FK)
    - email, passwordHash
    - role ('admin' | 'agent' | 'viewer')
    - createdAt, updatedAt

  leads:
    - id (UUID, PK)
    - tenantId (UUID, FK)
    - name (string, nullable)
    - phone (string, nullable)
    - email (string, nullable)
    - status ('new' | 'contacted' | 'qualified' | 'converted' | 'lost')
    - source (platform where first seen)
    - assignedTo (UUID, FK to users, nullable)
    - metadata (JSONB for custom fields)
    - createdAt, updatedAt

  conversations:
    - id (UUID, PK)
    - tenantId (UUID, FK)
    - leadId (UUID, FK)
    - platform (string)
    - platformUserId (string, indexed)
    - platformConversationId (string)
    - status ('active' | 'closed' | 'transferred')
    - lastMessageAt (timestamp)
    - createdAt

  messages:
    - id (UUID, PK)
    - tenantId (UUID, FK)
    - conversationId (UUID, FK)
    - direction ('inbound' | 'outbound')
    - content (TEXT)
    - attachments (JSONB)
    - sentAt (timestamp)
    - deliveredAt (timestamp, nullable)
    - readAt (timestamp, nullable)
    - metadata (JSONB, stores normalized + raw)

  events:
    - id (UUID, PK)
    - tenantId (UUID, FK)
    - leadId (UUID, FK)
    - type ('message_sent' | 'message_received' | 'status_changed' | 'workflow_started' | 'ai_called')
    - data (JSONB)
    - createdAt

  workflows:
    - id (UUID, PK)
    - tenantId (UUID, FK)
    - name (string)
    - trigger (JSONB)
    - steps (JSONB)
    - status ('active' | 'paused' | 'archived')
    - createdAt, updatedAt

  workflow_executions:
    - id (UUID, PK)
    - tenantId (UUID, FK)
    - workflowId (UUID, FK)
    - leadId (UUID, FK)
    - status ('in_progress' | 'completed' | 'failed')
    - currentStep (integer)
    - variables (JSONB)
    - startedAt, completedAt

Indexes:
  - leads: (tenantId, status), (tenantId, createdAt)
  - conversations: (tenantId, platformUserId), (leadId)
  - messages: (conversationId, sentAt), (tenantId, sentAt)
  - events: (tenantId, leadId, createdAt)

Row-Level Security:
  - Enable RLS on all tables
  - Policy: WHERE tenantId = current_setting('app.tenantId')::uuid

Lead Lifecycle:
  1. First message from unknown user → Create lead (status: new)
  2. Bot responds → status: contacted
  3. Intent classifier detects interest → status: qualified
  4. Workflow completes → status: converted or lost

CRM Operations:

  Lead Writer:
    - Upsert lead on every inbound message
    - Update lastMessageAt
    - Merge metadata if new info detected (name, email from message)

  Stage Tracker:
    - State machine: new → contacted → qualified → converted/lost
    - Triggered by:
      * Workflow completion
      * Manual agent update
      * AI classification (e.g., "not interested" → lost)

  Event Logger:
    - Log every interaction as event
    - Used for:
      * Analytics (conversion funnel)
      * Audit trail
      * Debugging

  Task Scheduler (via Workflow Engine):
    - Create delayed jobs for follow-ups
    - Example: If no reply in 24h → queue reminder message

Queries (via Prisma):
  - Get lead with conversations: prisma.lead.findUnique({
      where: { id, tenantId },
      include: { conversations: { include: { messages: true } } }
    })
  - Recent messages: prisma.message.findMany({
      where: { conversationId, tenantId },
      orderBy: { sentAt: 'desc' },
      take: 10
    })

Critical Rules:
  - ALWAYS filter by tenantId
  - Use transactions for multi-table updates
  - Store raw message in metadata for debugging
  - Index on high-cardinality columns (userId, conversationId)

LAYER 8: RATE LIMIT ORCHESTRATOR
---------------------------------
PURPOSE: Prevent API bans, throttle outbound messages per platform.

Per-Platform Limits:

  WhatsApp (Evolution API):
    - Official Meta limits apply if using Cloud API
    - Evolution API (unofficial): ~1000 messages/day safe
    - Strategy: 1 message/second burst, 500/day sustained

  Telegram:
    - 30 messages/second per bot
    - 20 messages/minute to same user
    - Strategy: Token bucket, 30 tokens/sec refill

  Discord:
    - 5 requests/second per bot
    - 50 messages/minute per channel
    - Strategy: Global + per-channel token buckets

  Meta (Messenger + Instagram):
    - Messenger: 300 messages/second per page
    - Instagram: 200 messages/hour per account
    - Constraint: 24-hour messaging window
    - Strategy: Strict token bucket, 200/hour for Instagram

  Teams:
    - No official limit, but 1000/day is safe
    - Strategy: Leaky bucket, 1/second

  Twitch:
    - Chat: 20 messages/30sec (verified bots: 100/30sec)
    - EventSub: No limit on incoming
    - Strategy: Token bucket, 20 tokens/30sec refill

  Twitter:
    - DMs: 500/day (free tier), 15000/day (paid)
    - Tweets: 300/3hours
    - Strategy: Daily quota tracking + rate limiter

Implementation (Bottleneck.js):

  const limiters = {
    whatsapp: new Bottleneck({
      reservoir: 500,           // 500 tokens
      reservoirRefreshAmount: 500,
      reservoirRefreshInterval: 24 * 60 * 60 * 1000, // daily
      maxConcurrent: 1,
      minTime: 1000             // 1/second
    }),
    telegram: new Bottleneck({
      reservoir: 30,
      reservoirRefreshAmount: 30,
      reservoirRefreshInterval: 1000, // 30/sec
      maxConcurrent: 5
    }),
    instagram: new Bottleneck({
      reservoir: 200,
      reservoirRefreshAmount: 200,
      reservoirRefreshInterval: 60 * 60 * 1000, // 200/hour
      maxConcurrent: 1,
      minTime: 18000            // ~200/hour = 1 every 18s
    })
    // ... other platforms
  }

  async function sendWithRateLimit(platform, sendFn) {
    return limiters[platform].schedule(() => sendFn())
  }

Redis Token Bucket (alternative):

  Key: ratelimit:{platform}:{identifier}
  Value: remaining_tokens
  Commands:
    - DECR: Take a token
    - TTL + EXPIRE: Refill tokens on interval
    - If tokens <= 0: Reject or queue

Backpressure Handling:
  - If rate limit hit → pause queue consumption
  - Resume when tokens refill
  - Don't retry immediately (causes thundering herd)

Monitoring:
  - Track: tokens_used, tokens_remaining, rate_limit_hits
  - Alert: If hitting limit >10 times/hour

Critical Rules:
  - NEVER bypass rate limiter
  - Per-tenant rate limits separate from platform limits
  - Log all rate limit hits for capacity planning
  - Graceful degradation (queue, don't drop)

---

LAYER 9: RESPONSE ROUTER
-------------------------
PURPOSE: Send final response back to user via correct platform.

Flow:
  1. Receive response from worker (unified format)
  2. Denormalize to platform-specific payload (use Layer 3 renderer)
  3. Apply rate limiter (Layer 8)
  4. Call platform adapter (Layer 1)
  5. Log delivery status
  6. Update message record in DB

Response Schema (unified):
  {
    tenantId: string
    platform: string
    userId: string
    conversationId: string
    message: {
      text: string
      attachments?: Array<{ type, url }>
      buttons?: Array<{ label, payload }>
      quickReplies?: Array<{ label, payload }>
    }
    metadata: {
      replyTo?: string
      notification?: boolean
    }
  }

Platform Selector:
  - Route to correct adapter based on response.platform
  - Fallback: If adapter unavailable → queue for retry

Format Converter (Outbound Renderer):
  - Use Layer 3 normalizer's outbound renderer
  - Platform-specific transformations:
    * WhatsApp: buttons → list message or reply buttons
    * Telegram: buttons → inline keyboard
    * Discord: buttons → message components
    * Meta: buttons → quick replies or template buttons
    * Teams: buttons → Adaptive Card actions

Send Executor:
  - Wrap in rate limiter: sendWithRateLimit(platform, () => adapter.send())
  - Retry on transient errors (network, 5xx)
  - No retry on permanent errors (invalid recipient, blocked user)

Delivery Tracker:
  - Update messages table:
    * deliveredAt: when API returns success
    * failedAt: if permanent failure
    * errorReason: if failed
  - Platform-specific receipts:
    * WhatsApp: Delivery + read receipts via webhook
    * Telegram: No read receipts
    * Discord: No delivery confirmation
    * Meta: Delivery + read receipts via webhook

Error Handling:
  - 429 Rate Limit: Back off, retry after delay
  - 4xx Client Error: Log, don't retry, alert
  - 5xx Server Error: Retry up to 3 times
  - Network Error: Retry with exponential backoff

Monitoring:
  - Metrics: messages_sent, send_latency, send_errors
  - Alert: Error rate >5%, latency >5s

Critical Rules:
  - ALWAYS denormalize before sending
  - Log every send attempt
  - Handle platform-specific errors gracefully
  - Update DB even on failure (for debugging)

6. MESSAGE FLOW (END-TO-END)
================================================================================

INBOUND MESSAGE FLOW
--------------------

1. Platform sends webhook POST
   - Example: WhatsApp user sends "Hello"
   - Evolution API sends POST to /gateway/whatsapp

2. Gateway receives, validates
   - Verify HMAC signature
   - Extract tenantId from API key in path/header
   - Check idempotency (Redis: idempotency:whatsapp:{msgId})
   - If duplicate: return 200 OK, skip processing
   - If new: continue

3. Normalize message
   - Parse raw webhook payload
   - Extract: userId, text, attachments, etc.
   - Store in unified format
   - Set idempotency key: SET idempotency:whatsapp:{msgId} "processed" EX 86400

4. Queue message
   - Push to BullMQ: whatsapp-messages queue
   - Job payload: { tenantId, platform: 'whatsapp', normalizedMessage }
   - Return 200 OK to platform (total time <3s)

5. Worker picks up job
   - BullMQ worker pulls from whatsapp-messages
   - Load session: GET session:{tenantId}:{userId}
   - Load context: LRANGE context:{tenantId}:{userId} 0 9

6. Intent classification
   - Check message text against regex patterns
   - Example: "price" → intent:pricing, confidence: 0.9
   - If confidence >0.8: skip AI

7. Rule engine check
   - Match intent:pricing → rule: send pricing message
   - Rule handled: yes
   - Response: "Our plans start at $99/month"

8. Update CRM
   - Upsert lead in PostgreSQL
   - Log event: { type: 'message_received', leadId, data: { message } }
   - Update conversation.lastMessageAt

9. Update session
   - SET session:{tenantId}:{userId} { ...session, lastMessageAt: now() }
   - LPUSH context:{tenantId}:{userId} { role: 'user', content: 'price' }

10. Queue response
    - Push to response queue with normalized response
    - Response router picks up

11. Rate limit check
    - Bottleneck checks whatsapp limiter
    - Tokens available: proceed
    - If no tokens: wait or queue

12. Send via platform adapter
    - Denormalize to WhatsApp format
    - POST to Evolution API: /message/sendText/{instanceId}
    - Store sent message in DB

13. Update delivery status
    - On success: UPDATE messages SET deliveredAt = now()
    - Log metrics: messages_sent++

OUTBOUND MESSAGE FLOW (SCHEDULED/WORKFLOW-TRIGGERED)
-----------------------------------------------------

1. Workflow engine triggers delayed job
   - Example: "Send follow-up in 24 hours if no reply"
   - BullMQ delayed job: scheduled-messages queue

2. Job executes at scheduled time
   - Worker picks up job
   - Check conversation: has user replied?
   - If yes: cancel
   - If no: proceed

3. Generate message
   - Use workflow template: "Hi {name}, just following up..."
   - Inject variables from lead data

4. Continue from step 9 above (update session, queue response, send)

7. RATE LIMITING STRATEGY (DETAILED)
================================================================================

See Layer 8 for implementation details.

Additional considerations:

Per-Tenant Rate Limits:
  - Free tier: 1000 messages/day total
  - Pro tier: 10K messages/day total
  - Enterprise: Unlimited

  Implementation:
    - Redis counter: ratelimit:tenant:{tenantId}:daily
    - INCR on every message processed

    - If counter > limit: Reject with 429, inform tenant

Burst vs Sustained:
  - Allow short bursts (e.g., 10 messages in 1 second)
  - But enforce daily/hourly caps for sustained load
  - Use token bucket (burst) + quota tracking (sustained)

Graceful Degradation:
  - If platform rate limit hit: Queue message, retry later
  - If tenant quota exceeded: Send email, upgrade prompt
  - Never drop messages silently

Monitoring:
  - Track per-tenant usage in billing table
  - Alert tenant at 80%, 90%, 100% of quota
  - Auto-upgrade flow: Prompt to upgrade plan

Testing Isolation:

  - Clean up test data after runs
  - Never pollute production tenants

Critical Rules:
  - NEVER allow tenantId in user input
  - ALWAYS derive from authenticated credentials
  - Test with 2+ tenants in every feature
  - Audit queries for missing tenantId filters

  - Worker pods: 1 CPU → 4 CPU

Horizontal Scaling:

  - Worker pods: 5 → 100 replicas (HPA on queue depth)

Database Scaling:
  - Read replicas: 1 → 3
  - Connection pooling: PgBouncer (500 connections)
  - Partitioning: messages table by month
  - Archival: Move old data (>1 year) to S3

Queue Scaling:
  - Redis Cluster: Shard by key prefix
  - Multiple worker pools: One per platform
  - Priority queues: VIP users separate queue

Caching Strategy:
  - L1: Application cache (in-memory, per pod)
  - L2: Redis cache (shared)
  - L3: CDN for static assets

Cost Optimization:
  - Spot instances for workers (up to 70% savings)
  - Reserved instances for database (up to 40% savings)
  - Auto-scale down during low traffic (nights, weekends)

Performance Targets:
  - API latency: p95 <200ms
  - Message processing: <5s end-to-end
  - Queue depth: <1000 jobs per worker
  - Database queries: <100ms p95

Capacity Planning:
  - 1 worker: ~100-200 messages/minute
  - 10 workers: ~1000-2000 messages/minute
  - 100 workers: ~10K-20K messages/minute

Bottleneck Analysis:
  - If queue growing: Scale workers
  - If DB slow: Add read replicas or optimize queries
  - If Redis slow: Scale to cluster

Global Expansion (Future):
