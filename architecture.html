System 1: CrmV2 (AI-Powered Bot SaaS)

System Overview

CrmV2 is a production-ready, multi-tenant SaaS platform enabling businesses to deploy AI-powered bots across 8 messaging platforms. It normalizes all incoming payloads, queues them, and processes them through an intelligent worker pipeline (Intent → Rules → Workflow → AI) with powerful CRM data persistence.

Technology Stack

Node.js 20+ TypeScript 5.3+ Express.js BullMQ 5.x Socket.IO PostgreSQL 15+ Prisma 5.x Redis 7+

Supported Platforms

  • WhatsApp: via Evolution API (Docker)
  • Telegram: tdlib/telegram-bot-api (Docker)
  • Discord: discord.js SDK
  • Meta (FB + IG): Direct Graph API Webhooks
  • Microsoft Teams: botbuilder-js
  • Twitch: twurple SDK (EventSub + IRC)
  • Twitter/X: twitter-api-v2

Layer-by-Layer Architecture

Layer 0: Multi-Tenancy & Isolation

Every request is strictly scoped to a tenant. No leaks permitted.

  • Tenant Router: Extracts tenantId from API key/JWT.
  • PostgreSQL RLS: Row-Level Security ensures queries are isolated via tenantId = current_setting('app.tenantId').
  • Redis Isolation: Keys are prefixed, e.g., cache:tenant:{tenantId}.

Layer 1: Platform API Adapters

Stateless adapters that handle specific API quirks and routing without polluting the core business logic.

Layer 2: Webhook Gateway

The single secure entry point handling incoming traffic.

  • Signature Validator: HMAC-SHA256, Ed25519 signatures validated on the fly.
  • Idempotency Layer: Uses Redis (idempotency:{platform}:{messageId}) with a 24h TTL to block duplicate webhook delivery.

Layer 3: Message Normalizer

A bidirectional parser. Translates raw, chaotic webhook payloads into a unified NormalizedMessage schema internally. It also translates outbound system responses back into platform-specific rich text and buttons.

Layer 4: Queue Layer (BullMQ)

Survives API spikes by decoupling receiving from processing. Utilizes per-platform queues (e.g., whatsapp-messages) with exponential backoff retries and Dead Letter Queues (DLQ).

Layer 5: Worker Pool

Stateless Node processes executing the intelligence pipeline asynchronously:

  1. Session Manager: Loads 24h conversational context from Redis.
  2. Intent Classifier: Fast NLP matching.
  3. Rule Engine: Hardcoded tenant static rules (e.g., "reply with $99 if intent is pricing").
  4. Workflow Engine: Handles multi-step state machine flows.
  5. AI Orchestrator: Fallback LLM logic.
  6. CRM Writer: Persists lifecycle changes to the database.

Layer 6: AI Orchestration Layer

Contains Cost Controllers (tracking token limits for Free/Pro tiers), a Model Router (tier 1: GPT-3.5, tier 2: GPT-4, tier 3: Claude), Prompt Builders, and Guardrails (PII scrubbing).

Layer 7: CRM Integration

PostgreSQL handles the business logic objects: Leads, Conversations, Messages, Events, Workflows, and Billing Usage.

Layer 8: Rate Limit Orchestrator

Prevents API bans and throttles outbound messages per platform using token buckets (e.g., 30 msgs/sec for Telegram, 500/day sustained for WhatsApp).

Layer 9: Response Router

Denormalizes unified responses to platform-specific payloads, executes sending wrapped in rate limits, and updates final DB delivery statuses.

End-to-End Message Flow

// 1. INBOUND FLOW
1. Platform Webhook ➜ Gateway
2. Gateway verifies HMAC ➜ Extracts Tenant ➜ Checks Idempotency (Redis)
3. Normalizer creates `NormalizedMessage` ➜ Sets Idempotency Flag
4. Pushed to BullMQ `whatsapp-messages` ➜ Return 200 OK
5. Worker pulls Job ➜ Loads Redis Session
6. Intent Classifier ➜ Rules Engine ➜ Workflow Engine ➜ AI
7. Update Prisma CRM (Lead, Conv, Messages) ➜ Update Redis Session
8. Queue Outbound Response

// 2. OUTBOUND FLOW
1. Response Router ➜ Rate Limit Check (Bottleneck.js)
2. Denormalize to target platform format
3. Send to API (e.g., Evolution API)
4. Update DB message status to `deliveredAt`

Infrastructure & Observability

Designed for mission-critical tracing and uptime tracking.

  • Logging: Pino (JSON structured logging with masked PII).
  • Metrics: Prometheus scraping counters (messages_sent) and histograms (queue_time).
  • Dashboards: Grafana visualization of queue depth and API latency.
  • Tracing: OpenTelemetry (W3C Trace Context) tracking payloads through the microservices.
  • Errors: Sentry auto-capturing unhandled promise rejections.

Security & Scaling Strategy

System scales automatically via Kubernetes HPA and protects data tightly.

  • Auth: JWTs (RS256) for users, HMAC-SHA256 hashed API Keys for webhooks.
  • Deployment: Kubernetes (EKS/GKE) with CI/CD GitHub Actions. Blue-Green deployments with instant rollbacks.
  • Scaling Horizontal: Workers scale 5 → 100 based on BullMQ depth. API scales 3 → 50 pods on CPU.
  • Scaling DB: PgBouncer connection pooling and Read Replicas.
